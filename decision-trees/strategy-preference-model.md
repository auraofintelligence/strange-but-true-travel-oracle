# Strategy Preference Model

Use this as a smart-dataset shape for modelling mixed preferences, plans, constraints, strategy and serendipity.

It is not the oracle. It is one way to structure data for the future oracle layer.

## Model Bands

- `no-go` - blocked by safety, law, entry, consent, money, health, privacy, or timing.
- `pause` - possible later, but the current model is unstable.
- `watch` - useful signal, not enough action evidence.
- `prepare` - promising, with missing checks.
- `candidate` - worth active comparison.
- `private` - keep in the dataset, do not publish.

## Preference Lanes

These lanes can be weighted differently for each strategy run:

- safety and consular viability
- visa and legal feasibility
- budget and route feasibility
- timing, rest and recovery
- serendipity and spontaneity
- romance and intercultural relationships
- Strange But True work value
- Shared Table and cultural context
- Musicverse and creative signal
- contacts, invitations and opportunity
- pattern and planning signal

If a numeric model is useful, normalise it inside the dated `strategy-run-{date}.md` file instead of pretending the same fixed score fits every country, mood, season, relationship field, budget state or invitation.

## Pattern Signal Rules

Pattern signal can add attention, not override danger.

Give strong pattern weight only when the signal produces a practical check, useful timing caution, better question, or safer next action.

## Minimum Gates

Even with strong preference weight:

- official safety, legal entry, health, money, consent, or privacy blockers can stop the move.
- romance/intercultural context must not treat relationship possibility as a reason to move unless consent, law, local norms and power differences are clear.
- Shared Table/cultural context must not publish any table story without protocol, consent and privacy review.
- insurance belongs only in legal, entry, venue, transport, visa or client-framework compliance checks.
